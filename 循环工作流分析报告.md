## 五、节点内容清除机制

### 5.1 清除时机

节点内容清除发生在 **分卷切换时**，具体在以下场景：

1. **章节正文生成完成后**，检测到分卷终止章触发
2. **循环节点执行时**，判断继续循环后重置节点状态

### 5.2 清除的节点类型

```typescript
nodesRef.current = nodesRef.current.map(n => {
  const typeKey = n.data.typeKey;
  // 清除这些创作类节点的输出
  if (['worldview', 'characters', 'plotOutline', 'outline'].includes(typeKey)) {
    return { ...n, data: { ...n.data, outputEntries: [] } };
  }
  // 重置章节节点的目标分卷
  if (typeKey === 'chapter' && nextVolumeName) {
    return { ...n, data: { ...n.data, targetVolumeId: '', targetVolumeName: nextVolumeName } };
  }
  // 重置创作信息节点状态
  if (typeKey === 'creationInfo') {
    return { ...n, data: { ...n.data, status: 'pending' } };
  }
  return n;
});
```

### 5.3 清除的内容

| outline | outputEntries | 大纲输出 |
| chapter | targetVolumeId, targetVolumeName | 章节目标分卷信息 |
| creationInfo | status | 重置为pending状态 |

## 六、严重Bug分析

### Bug 1：分卷名称严重不一致（Critical）

**问题描述**：
工作流中存在多套分卷命名系统，且互相不匹配：

1. **分卷规划器使用的名称**（8个分卷）：
   - 祭面之卷、绣龙之卷、烟云之卷、旌旗之卷、风眼之卷、壁观之卷、剑脊之卷、龙甲之卷

2. **multiCreateFolder使用的名称**（8个分卷）：
   - 祭舞之面、皇权图腾、吞云之斗、旌旗之纹、风暴之眼、深宫之绘、灵脉之锁、铁衣之扣

3. **循环指令配置使用的名称**（与分卷规划器一致）：
   - 祭面之卷、绣龙之卷...

**影响**：
- 文件夹创建时使用 `folderName`（如"祭舞之面"）
- 循环指令告知AI当前是"祭面之卷"
- 分卷切换时查找"祭面之卷"但实际卷名是"祭舞之面"
- 导致分卷匹配失败，章节可能创建在错误的分卷下

**代码证据**：

```json
  { "volumeName": "祭舞之面", "folderName": "祭舞之面", ... },
  { "volumeName": "皇权图腾", "folderName": "皇权图腾", ... },
  ...
]
```

循环指令配置（node-1775363003178）：
```json
"globalLoopInstructions": [
  { "content": "现在进行的是祭面之卷，共4章。第1章到第4章。" },
  { "content": "现在进行的是绣龙之卷，共5章。第1章到第5章。" },
  ...
]
```

分卷规划器输出（node-1775384775315）：

```json
"volumes": [
  { "volumeName": "祭面之卷", "folderName": "祭面之卷", ... },
```

### Bug 3：章节编号跨卷连续（High）

**问题描述**：
从contextSnapshot分析：
```json
"volumeEndChapters": [
  { "volumeId": "vol_idx_0", "endChapterTitle": "第7章" },
  { "volumeId": "vol_idx_1", "endChapterTitle": "第7章" },
  ...
]
```

每卷终止章都是"第7章"，说明：
- 系统期望每卷7章结束
- 但分卷规划只规划了4-5章
- 循环指令也是4-5章

**问题**：
- 循环指令和分卷规划都说第1卷是4章
- 但终止章检测却要到第7章
- AI实际只会生成4-5章（因为循环指令和规划这么说）
- 终止章检测永远等不到"第7章"

### Bug 4：循环节点与分卷切换逻辑冲突（High）

**问题描述**：
工作流同时存在两种分卷切换机制：

1. **循环节点（loopNode）**：循环8次，每次重置节点重新执行
2. **分卷终止章检测**：检测到特定章节标题后触发切换

**冲突点**：
```typescript
// 循环节点在第8次循环后完成
if (currentLoopIndex < loopConfig.count) {
  // 继续循环，重置节点
  i = targetIndex - 1;  // 跳回
} else {
  // 循环完成
}
```

但同时：
```typescript
// 章节生成时检测分卷终止章
const endChapterCheck = workflowManager.checkVolumeEndChapter(title, currentVolumeIndex);
if (endChapterCheck && endChapterCheck.shouldSwitchVolume) {
  // 触发分卷切换
}
```

**问题**：
- 循环节点在第8次迭代后退出循环
- 但如果分卷终止章检测在第4-5章就触发切换
- 切换会创建新分卷、重置节点
- 这与循环节点的逻辑可能冲突

### Bug 5：循环配置count=8但分卷规划4-5章/卷（Medium）

**问题描述**：

```typescript
let foundVolumes: any[] = [];
for (let j = 0; j < i; j++) {
  const prevNode = nodesRef.current.find(n => n.id === sortedNodes[j].id) || sortedNodes[j];
  if (prevNode.data.typeKey === 'saveToVolume' && prevNode.data.volumes) {

  // 使用从 saveToVolume 节点获取的完整分卷列表（强制覆盖，确保名称一致）
  volumeFolderConfigs = foundVolumes.map((vol: any, idx: number) => ({
```

### 8.3 修复分卷切换时机错误 (Critical Bug)

**问题**：终止章节号设置为4时，在第4章**开始创作**时就切换分卷了，而不是第4章**完成**后。

**根本原因**：多处判断使用了 `>=` 而不是 `>`：
- `currentChapterNum >= endChapterNum` → 第4章开始（4>=4）就触发
- 正确应该是 `currentChapterNum > endChapterNum` → 第4章完成（第5章开始时）才触发

**修复的文件**：

| 文件 | 位置 | 修改 |
|------|------|------|
| useWorkflowEngine.ts | 第2417行 | `>=` 改为 `>` |
| WorkflowManager.ts | 第496行 | `>=` 改为 `>` |
| WorkflowManager.ts | 第519行 | `>= endChapter + 1` 改为 `> endChapter` |
| WorkflowManager.ts | 第597行 | `>=` 改为 `>` |

**修复后的逻辑**：
```typescript
// 修复前：第4章开始就切换（错误）
if (endChapterNum > 0 && currentChapterNum >= endChapterNum) { ... }

// 修复后：第4章完成（第5章开始）才切换（正确）
if (endChapterNum > 0 && currentChapterNum > endChapterNum) { ... }
```

### 8.4 修复分卷进度未更新问题 (Critical Bug)

**问题**：第4章完成后，分卷切换正确发生，但"分卷创作信息"节点仍显示"第1卷"而不是"第2卷"。

**根本原因**：`nextVolIdx` 的条件判断有误：
```typescript
// 原代码
if (nextVolIdx >= 0 && nextVolIdx < volumePlans.length) {
  workflowManager.setCurrentVolumeIndex(nextVolIdx);
}
```

当 `volumePlans.length = 1`（只有1卷配置）但 `nextVolIdx = 1`（需要切换到第2卷）时：
- `nextVolIdx >= 0` = `true`
- `nextVolIdx < volumePlans.length` = `1 < 1` = `false`
- 结果：条件不满足，`setCurrentVolumeIndex` 不会被调用！

**修复方案**：移除 `nextVolIdx < volumePlans.length` 条件限制

```typescript
// 修复后：优先使用计算出的索引
if (nextVolIdx >= 0) {
  workflowManager.setCurrentVolumeIndex(nextVolIdx);
}
```

**修复的文件**：`useWorkflowEngine.ts` 第2507行

### 8.5 修复分卷切换后 Sets 仍写入错误卷的问题 (Critical Bug)

**问题**：分卷正确切换，但世界观、粗纲、大纲、灵感集、角色集仍储存到第一卷的文件夹中。

**根本原因**：`currentWorkflowFolder` 是执行引擎的局部变量，分卷切换时只更新了 `workflowManager` 的状态，没有更新创作节点的 `folderName` 属性。Sets 按 `folderName` 组织，所以内容仍写入旧卷。

**修复方案**：在分卷切换时，更新所有创作节点的 `folderName` 属性为新的分卷名称：

```typescript
// 清除创作类节点的输出，以便下一卷重新生成
nodesRef.current = nodesRef.current.map(n => {
  const typeKey = n.data.typeKey;
  if (['worldview', 'characters', 'plotOutline', 'outline'].includes(typeKey)) {
    // 更新节点的 folderName 为新的分卷名称，确保 Sets 写入正确的卷
    const updatedData = { ...n.data, outputEntries: [] };
    if (nextVolumeName) {
      updatedData.folderName = nextVolumeName;
    }
    return { ...n, data: updatedData };
  }
  ...
});
// 更新 currentWorkflowFolder（局部变量），确保后续节点使用正确的分卷
currentWorkflowFolder = nextVolumeName;
```

**修复的文件**：`useWorkflowEngine.ts` 第2527-2548行
