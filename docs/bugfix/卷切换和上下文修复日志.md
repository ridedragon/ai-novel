## Bug 1: 创作信息一直停留在第一卷

### 问题描述

当工作流通过循环节点（loopNode）切换到第二卷时，创作信息节点（creationInfo）仍然显示第一卷的分卷信息，不会随分卷切换更新当前分卷标题和进度。

### 问题分析

在 `useWorkflowEngine.ts` 的循环节点（loopNode）循环继续逻辑中（约第 1025-1050 行），`resetNodeData` 函数重置节点状态时，对于 `creationInfo` 类型的节点仅设置了 `status: 'pending'`，但**没有清空其 `outputEntries`**。

由于创作信息节点的输出内容（包含"当前分卷"、"分卷进度"等信息）是在节点首次执行时生成并保存到 `outputEntries` 中的，当循环切换卷时：

1. 节点状态被重置为 `pending`
2. 但 `outputEntries` 中仍然保留着第一卷的创作信息
3. 同时 `buildDynamicContext` 函数会收集所有前置节点的 `outputEntries` 作为上下文
4. 旧的创作信息作为上下文被传递给后续节点

关键代码（第 1042-1045 行）：
```typescript
// 重置创作信息节点
if (typeKey === 'creationInfo') {
  updates.status = 'pending';
  // 问题：没有清空 outputEntries，导致旧的卷信息被保留
}
```

### 修复方案

在循环节点继续时，清空创作信息节点的 `outputEntries`，确保下一卷开始时创作信息节点会重新生成新的分卷信息：

```typescript
// 重置创作信息节点
if (typeKey === 'creationInfo') {
  updates.status = 'pending';
  updates.outputEntries = [];  // 修复：清空旧的创作信息，让下一卷重新生成
}
```

## Bug 2: 分卷创作完成没有大小总结的章节没有触发总结

### 问题描述

当分卷的章节数量较少（如只有1-2章）时，该卷完成后不会自动生成大总结，导致后续卷的创作缺少对前卷的回顾。

### 问题分析

总结生成机制依赖于 `AutoWriteEngine` 的 `onChapterComplete` 回调和 `SummaryManager`。关键问题在于：

1. **总结触发条件依赖步长**：小总结和大总结的生成需要达到配置的 `consecutiveChapterCount`（通常设置为 3 或 6）
2. **卷切换时的总结机制**：在 `useWorkflowEngine.ts` 的分卷切换逻辑中（第 2604-2783 行），`shouldPauseForVolumeSwitch` 会返回 `true` 暂停工作流等待外部处理
3. **但暂停后没有对当前卷的残余章节触发总结检查**：如果当前卷的章节数不足步长值，这些章节永远不会触发总结

关键代码路径：
- `useWorkflowEngine.ts` 第 2542-2783 行：`onChapterComplete` 回调中的分卷切换逻辑
- `auto-write/index.ts` 第 150-184 行：分卷切换时的强制总结逻辑
- `auto-write/index.ts` 第 756-769 行：创作完成时的最终总结触发

第 756-769 行的收尾总结逻辑存在条件限制：
```typescript
// 核心改进：全自动创作彻底完成前，对最后一章触发一次"强制收尾"总结检查
const lastBatchItem = startIndex > 0 ? outline[startIndex - 1] : null;
if (lastBatchItem && this.isRunning) {
  const lastChapter = (this.novel.chapters || []).find(
    c => c.title === lastBatchItem.title &&
      ((targetVolumeId && c.volumeId === targetVolumeId) || (!targetVolumeId && !c.volumeId)),
  );
  if (lastChapter) {
    if (onChapterComplete) await onChapterComplete(lastChapter.id, lastChapter.content, this.novel, true);
  }
}
```

这个收尾总结只在 `this.isRunning` 为 `true` 时触发，但在某些异常场景下可能不会正确执行。

### 修复方案

确保在以下场景都触发总结检查：

1. **卷切换时**：在分卷切换逻辑完成后，增加对当前卷最后一个实际章节的总结检查
2. **单卷章节不足步长时**：即使章节数不满足常规总结步长，卷完成时也应强制触发总结

修复逻辑：在 `useWorkflowEngine.ts` 的分卷切换处理（第 2777-2780 行）之后，增加对当前卷残余章节的总结检查：

```typescript
// 在 shouldPauseForVolumeSwitch 返回之前，确保对当前卷的最后一个章节触发总结检查
// 修复：即使章节数不足总结步长，卷完成时也应强制触发总结
const lastStoryChapter = (localNovel.chapters || [])
  .filter(c => {
    const cVolId = c.volumeId || '';
    return cVolId === (currentVolId || '') && (!c.subtype || c.subtype === 'story');
  })
  .pop();

if (lastStoryChapter) {
  terminal.log(`[VOLUME_SWITCH] Triggering summary check for last chapter of completed volume: ${lastStoryChapter.title}`);
  // 复用 onChapterComplete 来触发总结检查
  const summaryResult = await globalConfig.onChapterComplete?.(
    lastStoryChapter.id,
    lastStoryChapter.content,
    localNovel,
    true,  // forceFinal = true 强制触发总结检查
  );
  if (summaryResult?.chapters) {
    localNovel = summaryResult;
    await updateLocalAndGlobal(localNovel);
  }
}
```

## Bug 3: 没有切换到第二卷文件夹

### 问题描述

当工作流从第一卷切换到第二卷时，节点的 `folderName` 没有正确更新为第二卷对应的文件夹名称，导致新生成的内容仍然被存储到第一卷的文件夹中。

### 问题分析

在 `useWorkflowEngine.ts` 的循环节点（loopNode）循环继续逻辑中（约第 930-1076 行），存在以下问题：

1. **重复代码块**：第 966-989 行和第 975-989 行有完全相同的"如果还没找到，尝试从 localNovel.volumes 获取"代码块，这是冗余代码

2. **文件夹名称更新范围不完整**：在循环继续时，节点重置逻辑（第 1037-1040 行）只更新了部分节点的 `folderName`：
```typescript
// 更新节点的 folderName 为下一个分卷名称（适用于所有需要关联目录的节点）
if (nextFolderName && typeKey !== 'createFolder' && typeKey !== 'multiCreateFolder' && 
    typeKey !== 'loopNode' && typeKey !== 'loopConfigurator' && typeKey !== 'pauseNode') {
  updates.folderName = nextFolderName;
}
```

3. `nextFolderName` 的查找顺序问题：
   - 首先从 `volumePlans` 查找
   - 然后从 `multiCreateFolder` 配置查找
   - 但这两个查找逻辑返回的结果可能不一致

关键代码（第 939-998 行）的排查：
```typescript
// 尝试从 volumePlans 获取
const volumePlans = workflowManager.getVolumePlans();
if (volumePlans[nextVolumeIndex]) {
  nextVolumeConfig = volumePlans[nextVolumeIndex];
  nextFolderName = nextVolumeConfig.folderName || nextVolumeConfig.volumeName || '';
}

// 如果没找到，尝试从 multiCreateFolder 配置获取
if (!nextFolderName) {
  for (const n of nodesRef.current) {
    if (n.data.typeKey === 'multiCreateFolder' && n.data.volumeFolderConfigs) {
      const configs = n.data.volumeFolderConfigs as any[];
      if (configs[nextVolumeIndex]) {
        nextFolderName = configs[nextVolumeIndex].folderName || configs[nextVolumeIndex].volumeName || '';
        break;
      }
    }
  }
}
```

问题在于：`nextFolderName` 可能查找到空值，因为 `volumePlans` 在分卷切换时可能被清空或未正确更新。

### 修复方案

1. **删除重复代码块**：移除第 975-989 行的重复代码
2. **确保 `nextFolderName` 查找逻辑优先使用 `localNovel.volumes`**
3. **增加兜底逻辑**：当 `nextFolderName` 仍然为空时，使用下一卷的 `title` 作为文件夹名称

修复后的逻辑：
```typescript
// 如果还没找到，尝试从 localNovel.volumes 获取
if (!nextFolderName && localNovel.volumes && localNovel.volumes[nextVolumeIndex]) {
  nextFolderName = localNovel.volumes[nextVolumeIndex].title || '';
  nextVolumeId = localNovel.volumes[nextVolumeIndex].id;
}

// 兜底：使用分卷索引作为最后的回退方案
if (!nextFolderName) {
  nextFolderName = `第${nextVolumeIndex + 1}卷`;
  terminal.warn(`[LOOP_NODE] Using fallback folder name: ${nextFolderName}`);
}
```

同时确保 `multiCreateFolder` 节点的 `currentWorkflowFolder` 也随分卷切换更新（第 1423 行）：
```typescript
currentWorkflowFolder = volumeFolderConfigs[currentVolumeIndex]?.folderName || 
                        volumeFolderConfigs[currentVolumeIndex]?.volumeName || 
                        nextFolderName || '';
```

## Bug 4: 生成内容列表(Output Entries)没有被清空

### 问题描述

当工作流循环继续执行第二卷时，世界观、角色、粗纲等节点的 `outputEntries` 列表中仍然包含第一卷的生成内容。这些旧内容会被作为上下文发送给后续节点的 AI 请求，导致 AI 混淆两卷的信息。

### 问题分析

在 `useWorkflowEngine.ts` 的循环节点继续逻辑中（约第 1025-1030 行）：

```typescript
// 清除创作类节点的输出，以便重新生成
// 核心修复：不清空 outline 节点的 outputEntries，因为大纲是整个小说的规划，不应随分卷切换而丢失
if (['worldview', 'characters', 'plotOutline'].includes(typeKey)) {
  updates.outputEntries = [];
}
```

这里虽然清空了 `worldview`、`characters`、`plotOutline` 的 `outputEntries`，但存在以下遗漏：

1. **`outline` 节点不应该保留旧内容**：虽然注释说"大纲是整个小说的规划"，但实际上每卷可能需要独立的细纲
2. **`inspiration`（灵感）节点没有被清空**：灵感集也应该随分卷切换清空
3. **循环配置器节点（`loopConfigurator`）的 `globalLoopInstructions` 没有被正确传递到后续执行**

同时，在 `buildDynamicContext` 函数中（第 266-429 行），会收集所有前置节点的 `outputEntries` 并作为上下文消息发送给 AI：

```typescript
if (pNode.data.outputEntries && pNode.data.outputEntries.length > 0) {
  // ... 收集输出条目并构建上下文消息
  dynamicContextMessages.push({
    role: 'system',
    content: `【${title}】：\n${combinedContent}`,
  });
}
```

这意味着如果 `outputEntries` 没有被清空，第一卷的信息会一直作为上下文传递到第二卷。

### 修复方案

扩大清空 `outputEntries` 的节点范围，在循环节点继续时清空更多类型的节点输出：

```typescript
// 清除创作类节点的输出，以便重新生成
// 修复：扩大清空范围，包含世界观、角色、粗纲、大纲和灵感节点
if (['worldview', 'characters', 'plotOutline', 'outline', 'inspiration'].includes(typeKey)) {
  updates.outputEntries = [];
}
```

同时，在 `buildDynamicContext` 函数中增加过滤逻辑，确保只收集当前卷的输出条目：

```typescript
// 本卷模式修复：过滤掉隔离边界之前的节点输出
// 同时清空历史卷的输出条目收集
if ((globalConfig.contextScope === 'volume' || globalConfig.contextScope === 'currentVolume') && j < boundaryIndex) {
  continue;  // 跳过隔离边界之前节点的所有输出
}
```

## 修复总结

| Bug # | 问题 | 根因 | 修复位置 |
|-------|------|------|----------|
| 1 | 创作信息停留在第一卷 | `creationInfo` 节点的 `outputEntries` 未被清空 | loopNode 继续逻辑 (~1042行) |
| 2 | 分卷完成未触发总结 | 章节不足步长时收尾总结条件不满足 | onChapterComplete 回调 + AutoWriteEngine 收尾逻辑 |
| 3 | 没有切换到第二卷文件夹 | `nextFolderName` 查找逻辑不完整 + 重复代码 | loopNode 分卷解析逻辑 (~966-1000行) |
| 4 | Output Entries 未清空 | 清空范围不够大，遗漏了 outline 和 inspiration | loopNode 继续逻辑 (~1027行) |

## 影响范围

- **仅影响**: `src/components/Workflow/hooks/useWorkflowEngine.ts`
- **不影响**: 已生成的章节内容、文件夹物理结构、第一卷数据
- **修复方式**: 仅修改工作流引擎的状态重置逻辑，不改变数据存储结构
