# 从指定位置启动工作流卷归属错误与章节消失修复日志

## 修复日期
2026/4/8

## 问题描述

用户从“指定位置启动工作流”时，明明选择的是第二卷，但生成出来的章节却先显示在第一卷下。

随后在生成完成或自动保存触发后，该章节又会从章节列表中消失，看起来像是“先写进去了，后来又没了”。

## 日志现象复盘

从日志可以看到两个连续异常：

### 现象 1：指定第二卷启动，但正文被切回第一卷

日志中的关键信息：

```text
[WORKFLOW_START] Final state: volIdx=1, anchor=3614ba7a...
[ChapterNode] fVolId=3614ba7a...
[WorkflowManager] Checking split: 8 unprocessed rules for "第一章：无形之契" (globalIndex: 2, norm: 1)
[WorkflowManager]   Checking FIRST rule: "祭舞之面" (endChapter: 1, chapterTitle: 第2章)
[WorkflowManager]   endChapter check: globalIndex=2 > endChapter=1 = true
[WorkflowManager]   TRIGGERED by endChapter
[WorkflowManager] Active Volume Anchor set to: cd51b2aa...
[AutoWriteEngine] Switched targetVolumeId to cd51b2aa... for chapter: 第一章：无形之契
```

这说明：

1. 启动时工作流的目标卷锚点确实已经是第二卷
2. 但章节真正写入前，又被旧的分卷规则触发
3. 触发后 `Active Volume Anchor` 被改写成了更早的卷
4. 最终章节被保存到了错误分卷中，所以章节列表显示在第一卷下

---

### 现象 2：章节短暂出现后又消失

日志中的另一组关键信息：

```text
[STORAGE] 更新元数据: 《测试》
[STORAGE] 更新大纲: 《测试》
[STORAGE] 更新章节正文: ChapterID=1775638284904, 长度=0
[PERF] storage.saveNovels (原子化并行模式) ...
```

同时，整个运行过程中存在大量高频保存、自动保存、世界观/大纲更新、工作流状态保存并行发生的情况。

这意味着：

1. 章节已被加入当前内存态，章节列表会短暂显示
2. 但随后某次较晚触发的 `saveNovels`，拿到的却是较旧的 `novels` 快照
3. 旧快照把新章节所在的章节列表状态覆盖掉
4. 结果就是章节从章节列表中“消失”

---

## 根因分析

## 根因 1：从指定卷启动后，旧 `pendingSplits` 规则仍可回切到更早卷

### 触发机制

在 `src/utils/WorkflowManager.ts` 的 `checkTriggerSplit` 中，虽然当前逻辑已经只检查“第一个未处理规则”，但在“从指定卷/指定位置启动”的场景里，仍然存在一个特殊问题：

- 用户指定从第二卷开始
- 但 `pendingSplits` 里可能还保留着“第一卷结束后切到第二卷”或者更早卷的旧规则
- 当当前章节的 `globalIndex` 已经大于该规则的 `endChapter` 时，旧规则会立刻触发
- 这样就把刚刚设置好的目标卷锚点又改写回旧卷

### 本质问题

系统缺少一个“启动卷锁”。

也就是说：

- 用户已经明确指定“从第 N 卷开始”
- 但运行期的旧分卷规则没有意识到这是一个“锁定起点”的启动
- 所以依然会执行“切到更早卷”的逻辑

---

## 根因 2：`storage.saveNovels` 的防抖保存会被旧快照覆盖

### 触发机制

在 `src/utils/storage.ts` 中，`saveNovels` 使用了 300ms 防抖：

```ts
if (this._saveDebounceTimer) {
  clearTimeout(this._saveDebounceTimer);
}
this._pendingNovels = novels;
```

这段逻辑的风险在于：

- 短时间内会有很多不同来源的 `saveNovels(novels)` 调用
- 每次调用都直接把 `_pendingNovels` 整体替换成“本次传入的 novels”
- 如果后一次调用拿到的是“较旧的内存快照”，它仍然会覆盖前一次“较新的快照”
- 防抖结束时，真正落盘的是最后一次传入的那份旧数据

### 本质问题

当前防抖实现只保证“最后一次调用会保存”，但不保证“最后保存的是最新数据”。

在章节生成场景下，这会直接表现为：

- 新章节已进入最新内存态
- 后续某次旧快照保存把它覆盖掉
- 章节列表重新渲染后，该章节消失

---

## 修复方案

## 修复 1：为“从指定卷启动”增加启动卷锁，阻止旧规则回切更早卷

### 文件
`src/utils/WorkflowManager.ts`

### 修改内容

在 `WorkflowState` 中增加启动卷锁状态：

- `lockedStartVolumeId`
- `lockedStartVolumeIndex`

并新增以下方法：

- `lockStartVolume(volumeId?, volumeIndex?)`
- `clearStartVolumeLock()`
- `getStartVolumeLock()`

### 同时修改 `checkTriggerSplit`

在检查首个未处理规则时，新增启动卷锁判断：

- 如果当前是“指定卷启动”
- 且该规则对应的目标卷索引小于启动卷索引
- 则直接 suppress，不允许触发

核心效果：

1. 用户指定从第二卷启动后
2. 第一卷遗留规则即使满足 `globalIndex > endChapter`
3. 也不能再把当前执行回切到第一卷
4. 当前章节会继续落在用户指定卷

---

## 修复 2：启动工作流时显式锁定目标卷，并同步更新后续章节节点目标卷

### 文件
`src/components/Workflow/hooks/useWorkflowEngine.ts`

### 修改内容

在 `runWorkflow` 中：

1. 解析用户指定的 `targetVolumeId`
2. 找到对应的 `targetVolumeIndex`
3. 调用：
   - `workflowManager.lockStartVolume(...)`
   - `workflowManager.setActiveVolumeAnchor(...)`
   - `workflowManager.setCurrentVolumeIndex(...)`

并在“从指定位置开始”的恢复逻辑里：

- 如果用户已经指定目标卷，则不再允许恢复逻辑用旧节点状态覆盖当前目标卷
- 同时把 `startIndex` 之后的所有章节节点的：
  - `targetVolumeId`
  - `targetVolumeName`
  - `folderName`
  
统一改为用户当前选择的卷

### 核心效果

这样修复后，指定卷启动不再只是“设置一个临时 anchor”，而是变成：

- 启动态锁定
- 恢复态不覆盖
- 后续章节节点目标卷统一重写

避免出现“表面上选了第二卷，但章节节点自己还拿着旧卷配置”的问题。

---

## 修复 3：防抖保存改为“合并待保存快照”，不再让旧快照覆盖新章节

### 文件
`src/utils/storage.ts`

### 修改内容

新增了以下合并函数：

- `_mergePendingChapter`
- `_mergePendingNovel`
- `_mergePendingNovels`

并把原来的：

```ts
this._pendingNovels = novels;
```

改为：

```ts
this._pendingNovels = this._mergePendingNovels(this._pendingNovels, novels);
```

### 合并策略

#### 章节级合并
对同一个章节：

- 优先保留更长的 `content`
- 优先保留存在的 `volumeId`
- 优先保留更完整的 `versions`
- 保留较新的分析结果、编号信息和版本状态

#### 小说级合并
对同一本书：

- 按章节 `id` 合并章节列表
- 避免新章节被旧章节数组冲掉
- 合并卷信息
- 对设定、大纲、角色、灵感、资料库等数组，优先保留更完整的一份

### 核心效果

这样即使出现以下情况：

1. A 次保存拿到新章节的最新快照
2. B 次保存稍后触发，但传入的是较旧快照
3. 防抖阶段二者会先合并，而不是简单覆盖
4. 最终落盘仍能保住新章节和正确卷归属

这就修复了“章节先出现、后消失”的问题。

## 修复后的预期行为

修复后，从“指定位置启动工作流”并选择第二卷时：

1. 工作流启动后，卷锚点锁定在第二卷
2. 旧分卷规则不会再把当前执行错误切回第一卷
3. 新生成章节会正确挂到第二卷下
4. 即使过程中发生多次自动保存/大纲保存/世界观保存
5. 新章节也不会再因为旧快照覆盖而从章节列表中消失

## 建议验证步骤

### 场景 1：从第二卷起始节点启动
1. 选择一个已有分卷规划的小说
2. 在工作流中选择从中间某个章节节点启动
3. 指定目标卷为第二卷
4. 执行正文生成
5. 确认新章节出现在第二卷下，而不是第一卷

### 场景 2：观察生成后是否消失
1. 启动生成后，等待章节出现在列表中
2. 不要手动刷新，继续等待 5~20 秒，让自动保存多触发几轮
3. 确认章节不会消失，也不会被重新归类到错误卷

### 场景 3：带旧分卷规则的恢复执行
1. 使用包含多条 `pendingSplits` 历史规则的工作流
2. 从后续卷中间位置启动
3. 确认日志里不会再出现“被更早卷规则回切”的异常行为

---

## 备注

本次问题不是单一 bug，而是两个问题叠加造成的：

1. **运行期卷归属被旧规则抢回**
2. **保存期章节状态被旧快照覆盖**

因此用户看到的是一个复合现象：

- 先显示在错误卷
- 再从章节列表中消失

只有同时修复“卷路由”和“保存一致性”，问题才能真正闭环解决。
