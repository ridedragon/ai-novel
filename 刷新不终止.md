# 页面刷新重载工作流不终止实现方案

## 问题分析

当前项目中，工作流执行状态完全依赖于内存中的 `WorkflowManager` 单例。当页面刷新时，内存状态丢失，导致工作流执行被终止，用户需要重新启动工作流。

## 实现方案

### 1. 工作流状态持久化

#### 1.1 扩展 WorkflowManager 类

在 `WorkflowManager` 中添加状态持久化方法，将工作流执行状态保存到 IndexedDB。

```typescript
// src/utils/WorkflowManager.ts

import terminal from 'virtual:terminal';
import { Chapter, VariableBinding, WorkflowContextSnapshot, WorkflowGlobalContext } from '../types';
import { resolveMacros, MacroContext, MACRO_QUICK_REFERENCE } from '../components/Workflow/macros';
import { storage } from './storage';

export type WorkflowStatus = 'idle' | 'running' | 'paused' | 'failed';

interface WorkflowState {
  isRunning: boolean;
  isPaused: boolean;
  currentNodeIndex: number;
  activeWorkflowId: string | null;
  error: string | null;
  globalContext: WorkflowGlobalContext;
  totalVolumes: number; // 计划的总分卷数
  lockedStartVolumeId?: string;
  lockedStartVolumeIndex?: number;
}

// 新增：工作流执行状态接口
interface WorkflowExecutionState {
  runId: string | null;
  state: WorkflowState;
  timestamp: number;
}

const WORKFLOW_EXECUTION_KEY = 'workflow_execution_state';

class WorkflowManager {
  private state: WorkflowState = {
    isRunning: false,
    isPaused: false,
    currentNodeIndex: -1,
    activeWorkflowId: null,
    error: null,
    totalVolumes: 0,
    lockedStartVolumeId: undefined,
    lockedStartVolumeIndex: undefined,
    globalContext: {
      variables: {},
      executionStack: [],
    },
  };

  // 核心修复 (Bug 2): 引入全局执行 ID，用于识别并废弃过时的异步回调
  private currentRunId: string | null = null;

  private listeners: Set<StateListener> = new Set();
  private nodeUpdateListeners: Set<NodeUpdateListener> = new Set();

  constructor() {
    // 构造函数中恢复执行状态
    this.loadExecutionState();
  }

  // 新增：保存执行状态到 IndexedDB
  private async saveExecutionState() {
    try {
      const executionState: WorkflowExecutionState = {
        runId: this.currentRunId,
        state: this.state,
        timestamp: Date.now(),
      };
      await storage.set(WORKFLOW_EXECUTION_KEY, executionState);
      terminal.log(`[WorkflowManager] Execution state saved`);
    } catch (error) {
      terminal.error(`[WorkflowManager] Failed to save execution state: ${error}`);
    }
  }

  // 新增：从 IndexedDB 加载执行状态
  private async loadExecutionState() {
    try {
      const executionState = await storage.get<WorkflowExecutionState>(WORKFLOW_EXECUTION_KEY);
      if (executionState) {
        this.currentRunId = executionState.runId;
        this.state = executionState.state;
        terminal.log(`[WorkflowManager] Execution state loaded`);
        this.notify();
      }
    } catch (error) {
      terminal.error(`[WorkflowManager] Failed to load execution state: ${error}`);
    }
  }

  // 重写 start 方法，添加状态保存
  public start(workflowId: string, startIndex: number = 0, snapshot?: WorkflowContextSnapshot) {
    // 核心修复 (Bug 2): 生成并锁定本次运行的唯一 ID
    this.currentRunId = `run_${Date.now()}_${Math.random().toString(36).substr(2, 9)}`;
    terminal.log(
      `[WorkflowManager] Starting workflow: ${workflowId} at index ${startIndex} (RunID: ${this.currentRunId})`,
    );

    // 确定新的上下文逻辑
    let newContext: WorkflowGlobalContext;

    if (startIndex === 0) {
      // 1. 完全从头开始：彻底重置
      newContext = {
        variables: {
          loop_index: 1,
          batch_range: '',
        },
        executionStack: [],
        activeVolumeAnchor: undefined,
        pendingSplitChapter: undefined,
        pendingNextVolumeName: undefined,
        pendingSplits: [],
        volumePlans: [],
        volumeEndChapters: [],
      };
    } else if (startIndex > 0) {
      // 2. 恢复执行逻辑
      const memoryContext = this.state.globalContext;
      const hasMemory = Object.keys(memoryContext.variables).length > 2 || memoryContext.activeVolumeAnchor;

      if (hasMemory && this.state.activeWorkflowId === workflowId) {
        // 2a. 优先使用内存中的上下文 (最实时)
        terminal.log(`[WorkflowManager] Resuming with in-memory context.`);
        newContext = memoryContext;
      } else if (snapshot) {
        // 2b. 内存丢失或切换了工作流，尝试从持久化快照恢复
        terminal.log(`[WorkflowManager] Resuming with context snapshot recovery.`);
        newContext = {
          variables: snapshot.variables || { loop_index: 1 },
          executionStack: [],
          activeVolumeAnchor: snapshot.activeVolumeAnchor,
          pendingSplits: snapshot.pendingSplits || [],
          volumePlans: snapshot.volumePlans || [],
          volumeEndChapters: snapshot.volumeEndChapters || [],
        };
      } else {
        // 2c. 既无内存也无快照，维持现状
        newContext = memoryContext;
      }

      // 核心增强 (Bug 1 反馈修复)：如果恢复时没有锚点，但不是从头开始，尝试从全局变量回溯
      // 这能解决因快照未能及时保存导致的“分卷归类丢失”问题
      if (!newContext.activeVolumeAnchor && startIndex > 0) {
        terminal.warn(`[WorkflowManager] Anchor missing in snapshot. Will attempt back-tracing in engine.`);
      }
    } else {
      // 3. 正常的内存内继续执行
      newContext = this.state.globalContext;
    }

    this.state = {
      ...this.state,
      isRunning: true,
      isPaused: false,
      currentNodeIndex: startIndex,
      activeWorkflowId: workflowId,
      error: null,
      globalContext: newContext,
    };
    
    // 保存状态
    this.saveExecutionState();
    this.notify();
  }

  // 重写 pause 方法，添加状态保存
  public pause(index: number) {
    terminal.log(`[WorkflowManager] Pausing workflow at index ${index}`);
    this.state = {
      ...this.state,
      isRunning: false,
      isPaused: true,
      currentNodeIndex: index,
    };
    
    // 保存状态
    this.saveExecutionState();
    this.notify();
  }

  // 重写 stop 方法，添加状态保存
  public stop() {
    terminal.log(`[WorkflowManager] Stopping/Completing workflow (RunID reset)`);
    this.currentRunId = null;
    this.state = {
      ...this.state,
      isRunning: false,
      isPaused: false,
      currentNodeIndex: -1,
      totalVolumes: 0,
      lockedStartVolumeId: undefined,
      lockedStartVolumeIndex: undefined,
      globalContext: {
        variables: {
          loop_index: 1,
          batch_range: '',
        },
        executionStack: [],
        activeVolumeAnchor: undefined,
        pendingSplitChapter: undefined,
        pendingNextVolumeName: undefined,
        pendingSplits: [],
        volumePlans: [],
        volumeEndChapters: undefined,
      },
    };
    
    // 清除保存的状态
    storage.del(WORKFLOW_EXECUTION_KEY);
    this.notify();
  }

  // 重写 updateProgress 方法，添加状态保存
  public updateProgress(index: number) {
    if (this.state.currentNodeIndex !== index) {
      this.state.currentNodeIndex = index;
      
      // 保存状态
      this.saveExecutionState();
      this.notify();
    }
  }

  // 重写 updateContext 方法，添加状态保存
  public updateContext(updates: Partial<WorkflowGlobalContext>) {
    // 深度合并变量
    const newVariables = {
      ...this.state.globalContext.variables,
      ...(updates.variables || {}),
    };

    this.state = {
      ...this.state,
      globalContext: {
        ...this.state.globalContext,
        ...updates,
        variables: newVariables,
      },
    };
    
    // 保存状态
    this.saveExecutionState();
    this.notify();
  }

  // 其他方法保持不变...
}

export const workflowManager = new WorkflowManager();
```

#### 1.2 扩展 storage 工具

在 `storage.ts` 中添加对工作流执行状态的存储支持。

```typescript
// src/utils/storage.ts

// 新增工作流执行状态键
const WORKFLOW_EXECUTION_KEY = 'workflow_execution_state';

export const storage = {
  // 现有方法...

  // 新增：保存工作流执行状态
  async saveWorkflowExecutionState(state: any): Promise<void> {
    try {
      await set(WORKFLOW_EXECUTION_KEY, state);
      await saveToApi(WORKFLOW_EXECUTION_KEY, state);
    } catch (e) {
      console.error('Failed to save workflow execution state', e);
    }
  },

  // 新增：获取工作流执行状态
  async getWorkflowExecutionState(): Promise<any> {
    try {
      let state = await get<any>(WORKFLOW_EXECUTION_KEY);
      if (!state) {
        const remoteState = await fetchFromApi<any>(WORKFLOW_EXECUTION_KEY);
        if (remoteState) {
          state = remoteState;
          await set(WORKFLOW_EXECUTION_KEY, state);
        }
      }
      return state;
    } catch (e) {
      console.error('Failed to get workflow execution state', e);
      return null;
    }
  },

  // 新增：删除工作流执行状态
  async deleteWorkflowExecutionState(): Promise<void> {
    try {
      await del(WORKFLOW_EXECUTION_KEY);
      await deleteFromApi(WORKFLOW_EXECUTION_KEY);
    } catch (e) {
      console.error('Failed to delete workflow execution state', e);
    }
  },
};
```

## 实现可能性分析

### 技术可行性

1. **状态持久化**：使用 IndexedDB 存储工作流执行状态，这是现代浏览器支持的标准功能，容量大，性能好。

2. **状态恢复**：在 WorkflowManager 构造函数中加载保存的状态，确保应用启动时就能恢复工作流状态。

3. **异步任务管理**：使用 AbortController 管理异步任务，确保在页面刷新后能够正确处理正在执行的任务。

4. **实时状态更新**：在工作流状态变化时及时保存到 IndexedDB，确保状态的实时性。

### 实现复杂度

1. **低复杂度**：主要是添加状态持久化和恢复逻辑，不需要修改现有的工作流执行逻辑。

2. **高可靠性**：使用成熟的 IndexedDB 技术，确保数据的可靠存储。

3. **向后兼容**：不影响现有的工作流功能，只是添加了状态持久化和恢复能力。

## 总结

通过实现工作流状态的持久化和恢复机制，可以确保页面刷新重载时工作流不终止，提高用户体验。该方案技术可行，实现复杂度低，且对现有功能影响最小。

### 关键优势

1. **用户体验提升**：用户可以在页面刷新后继续之前的工作流执行，无需重新启动。

2. **可靠性增强**：即使在网络不稳定或浏览器崩溃的情况下，工作流状态也能得到保留。

3. **实现简单**：利用现有的 IndexedDB 存储机制，无需引入新的依赖。

4. **向后兼容**：不影响现有的工作流功能，只是添加了状态持久化和恢复能力。

### 潜在风险

1. **存储容量**：如果工作流执行状态过大，可能会占用较多的 IndexedDB 存储空间。

2. **状态同步**：在多标签页同时执行工作流时，可能会出现状态冲突。

3. **性能开销**：频繁的状态保存可能会对性能产生一定影响。

### 解决方案

1. **存储优化**：只保存必要的执行状态，定期清理过期的状态数据。

2. **状态同步**：使用 localStorage 的 storage 事件监听多标签页状态变化，确保状态一致性。

3. **性能优化**：使用防抖机制，避免过于频繁的状态保存操作。

通过以上方案，可以实现页面刷新重载时工作流不终止的功能，提高用户体验和系统可靠性。
